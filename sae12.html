Problème 1 — Le poste client ne recevait plus d’adresse IP automatiquement

Introduction

Dans le cadre de la SAE12, j’ai mis en place un serveur DHCP sur une machine Debian afin de distribuer automatiquement des adresses IP aux postes du réseau. Cette configuration permet de simplifier l’administration du réseau et d’éviter une configuration manuelle des machines clientes.

Après avoir configuré le serveur DHCP ainsi que le poste client, j’ai réalisé plusieurs tests réseau afin de vérifier le bon fonctionnement de l’attribution automatique des adresses IP.

Apparition du problème

Lors des tests de connexion, le poste client ne recevait plus d’adresse IP automatiquement depuis le serveur DHCP.

Après vérification avec la commande suivante :

ip a

aucune adresse IP valide n’était attribuée à l’interface réseau du client.

Figure 1 — Absence d’adresse IP sur le poste client.

Cette situation indiquait que le client ne parvenait plus à communiquer correctement avec le serveur DHCP.

Analyse du problème

Après plusieurs vérifications sur le serveur Debian, j’ai constaté que le service DHCP était arrêté.

La commande suivante :

systemctl status isc-dhcp-server

affichait l’état :

inactive (dead)

Figure 2 — Service DHCP arrêté.

Le problème ne provenait donc pas du poste client, mais du serveur DHCP qui ne distribuait plus les paramètres réseau.

Résolution du problème

Pour corriger cette erreur, j’ai modifié le fichier suivant :

/etc/default/isc-dhcp-server

Puis j’ai configuré l’interface réseau utilisée par le serveur DHCP :

INTERFACESv4="eth0"

Figure 3 — Configuration de l’interface réseau.

J’ai ensuite vérifié et complété la configuration du fichier DHCP :

/etc/dhcp/dhcpd.conf

en ajoutant le sous-réseau, la plage d’adresses IP ainsi que la passerelle par défaut.

Figure 4 — Configuration du fichier dhcpd.conf.

Après cette correction, j’ai redémarré le service DHCP :

sudo systemctl restart isc-dhcp-server

Puis j’ai vérifié son état :

systemctl status isc-dhcp-server

Le service affichait désormais :

active (running)

Figure 5 — Service DHCP actif après correction.

Après renouvellement de la configuration réseau du poste client, celui-ci a correctement reçu une adresse IP dynamique depuis le serveur DHCP.

Figure 6 — Attribution correcte d’une adresse IP.

Ce que cette situation m’a appris

  • Le fonctionnement d’un serveur DHCP sous Linux ;
  • L’importance des fichiers de configuration réseau ;
  • L’utilisation des services Linux avec systemctl ;
  • Les méthodes de diagnostic d’un problème d’attribution IP ;
  • L’importance des interfaces réseau dans les services DHCP.

Problème 2 — Le serveur Apache était inaccessible depuis le réseau

Introduction

Dans le cadre de la SAE12, j’ai installé et configuré un serveur web Apache sur une machine Debian afin d’héberger un service HTTP accessible depuis le réseau local.

Après l’installation du service, plusieurs tests ont été réalisés afin de vérifier l’accessibilité du serveur depuis une machine cliente.

Apparition du problème

Le serveur Apache fonctionnait correctement en local sur la machine Debian. Cependant, les postes clients du réseau ne parvenaient pas à accéder au site web hébergé.

Depuis le poste client, l’accès à :

http://192.168.1.62

échouait et la page web ne s’affichait pas.

Figure 1 — Échec d’accès au serveur Apache.

Analyse du problème

Après plusieurs vérifications réseau, j’ai constaté que le service Apache écoutait uniquement sur l’interface locale (127.0.0.1), ce qui empêchait les connexions provenant des autres machines du réseau.

Le problème ne provenait donc pas du client ni du réseau IP, mais de la configuration du serveur web Apache.

Résolution du problème

Pour corriger cette erreur, j’ai modifié la configuration d’écoute du serveur Apache afin qu’il accepte les connexions provenant du réseau local.

J’ai vérifié le fichier de configuration :

sudo nano /etc/apache2/ports.conf

La correction consistait à supprimer l’adresse 127.0.0.1.

Après modification de la configuration et redémarrage du service Apache :

sudo systemctl restart apache2

Le site web est devenu accessible depuis les autres machines du réseau.

Figure 2 — Accès réussi au serveur Apache.

Ce que cette situation m’a appris

  • Le fonctionnement d’un serveur web Apache ;
  • Les mécanismes d’écoute réseau sous Linux ;
  • Les tests client/serveur ;
  • Les méthodes de diagnostic d’accessibilité réseau ;
  • L’importance des paramètres réseau dans les services web.
